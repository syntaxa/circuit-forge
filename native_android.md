# Circuit Forge: подходы к портированию на Android

## 1) Что есть сейчас (и почему это важно для миграции)

Текущее приложение — SPA на `React + TypeScript + Vite` с чётким разделением на:

- UI-экраны в `components/*` (`ScreenSetup`, `ScreenWorkout`, `ScreenDatabase` и т.д.);
- бизнес-логику в `services/*` (`workoutGenerator`, `storageService`, `ttsService`, `wakeLockService`);
- типы и константы (`types.ts`, `constants.ts`).

### Сильные стороны для переноса

- `WorkoutGenerator` уже изолирован и почти не зависит от UI.
- Модель данных (`Exercise`, `Settings`, `WorkoutLog`) формализована.
- Есть автотесты (unit/component/integration), которые можно использовать как регрессионный контракт.

### Технические зависимости, которые не перенесутся «как есть»

- `localStorage` (нужна замена на Android-хранилище).
- Web Speech API (`speechSynthesis`) в `ttsService`.
- Web Wake Lock API + video fallback в `wakeLockService`.
- Навигация через browser history (`history.pushState`, `popstate`, `beforeunload`).

Итого: логика в целом переносима, но слой платформенных API нужно адаптировать.

## 2) Основные варианты портирования

## Вариант A — Android-обёртка над вебом (TWA/WebView/Capacitor)

### Идея

Оставить текущий React-код почти без изменений, упаковать его в Android-приложение.

### Подварианты

- **TWA (Trusted Web Activity)**: открывает PWA в Chrome внутри Android-приложения.
- **Capacitor**: WebView + bridge к нативным API.
- **Чистый WebView**: минимальная обёртка, но больше ручной инфраструктуры.

### Плюсы

- Максимально быстрый time-to-market.
- Минимальная стоимость запуска.
- Почти полный реюз текущего кода.

### Минусы

- UX и производительность ниже, чем у полноценного native.
- Ограничения/нестабильность на уровне веб-API (особенно TTS и wake lock).
- Сложнее добиться «магазинного» качества (Play Store expectations).

### Когда выбирать

- Нужен релиз Android за 2-4 недели для валидации гипотезы.
- Команда небольшая, приоритет — скорость и низкий бюджет.

---

## Вариант B — React Native (рекомендованный компромисс)

### Идея

Переписать UI на React Native, а бизнес-логику переиспользовать/вынести в shared TypeScript core.

### Что можно переиспользовать почти напрямую

- Типы (`types.ts`) и часть моделей.
- Алгоритм генерации тренировок (`workoutGenerator.ts`) после небольшой адаптации зависимостей.
- Базовые правила домена (ротация групп, ограничения по сложности, история).

### Что переписывается

- Все экранные компоненты (`components/*`) на RN-компоненты.
- `storageService`: на `AsyncStorage`, `MMKV` или SQLite.
- `ttsService`: на `react-native-tts`.
- `wakeLockService`: на `expo-keep-awake` или нативные Android flags.
- Навигация: на React Navigation (stack/modal patterns).

### Плюсы

- Существенно лучше мобильный UX, чем у WebView-подхода.
- Остаётся TypeScript-экосистема и знакомый React-подход.
- Быстрее и дешевле, чем полностью Kotlin-native.

### Минусы

- Всё равно нужна значимая перепись UI и платформенного слоя.
- Нужно аккуратно спроектировать shared-core, чтобы не утащить веб-зависимости.

### Когда выбирать

- Нужен «почти нативный» продукт на Android с перспективой iOS.
- Важен баланс между сроками, качеством и поддерживаемостью.

---

## Вариант C — Полностью нативный Android (Kotlin + Jetpack Compose)

### Идея

Собрать Android-приложение с нуля на Kotlin/Compose, доменную логику перенести в Kotlin.

### Плюсы

- Лучший UX/перформанс/контроль платформы.
- Нативные API (TTS, lifecycle, foreground behavior, audio focus, wake lock) интегрируются чисто.
- Проще соответствовать Android-гайдлайнам.

### Минусы

- Самая высокая стоимость и сроки.
- Переиспользование текущего TS-кода минимальное (в основном логика переписывается).
- Для iOS потребуется отдельная реализация (или KMP позже).

### Когда выбирать

- Android — ключевая платформа, нужен долгосрочный premium-уровень.
- Команда сильна в Kotlin/Android и готова к большей инвестиции.

---

## Вариант D — Kotlin Multiplatform core + Native UI

### Идея

Вынести доменную логику и модели в KMP-модуль, UI делать нативно (Compose/SwiftUI).

### Плюсы

- Максимально чистая архитектура для мультиплатформы в долгую.
- Реальный shared core между Android и iOS.

### Минусы

- Сложнее старт и выше архитектурный порог.
- Не лучший вариант для быстрого первого Android-релиза.

### Когда выбирать

- Есть план сразу на Android+iOS и зрелая команда/процессы.

## 3) Сравнение вариантов

| Критерий | A: Обёртка веба | B: React Native | C: Kotlin Native | D: KMP + Native UI |
|---|---|---|---|---|
| Скорость первого релиза | Очень высокая | Средняя | Низкая | Низкая |
| Стоимость старта | Низкая | Средняя | Высокая | Высокая |
| Качество Android UX | Среднее | Высокое | Очень высокое | Очень высокое |
| Реюз текущего кода | Очень высокий | Средний/высокий (core) | Низкий | Низкий/средний |
| Риск по платформенным API | Средний/высокий | Средний | Низкий | Низкий |
| Долгосрочная масштабируемость | Средняя | Высокая | Высокая | Очень высокая |

## 4) Рекомендация для Circuit Forge

Оптимальная стратегия для этого проекта: **двухшаговая**.

1. **Краткосрочно**: довести PWA и при необходимости сделать Android-обёртку для быстрого выхода на пользователей.
2. **Среднесрочно**: основной мобильный продукт строить на **React Native** с вынесением доменной логики в shared TS core.

Почему именно так:

- Уже есть хорошая доменная логика на TypeScript (особенно генератор и модель данных).
- Главные риски находятся в платформенных API и UI-слое, а не в домене.
- Такой путь даёт быстрый релиз и не запирает в слабом UX надолго.

## 5) Практический план миграции (для варианта B)

## Этап 0 — Подготовка доменного ядра

- Выделить модуль `core` без зависимости от React/DOM/window.
- Перенести туда:
  - типы домена;
  - `workoutGenerator`;
  - чистые правила валидации/формирования данных.
- Оставить только интерфейсы для инфраструктуры (`StoragePort`, `TtsPort`, `WakeLockPort`).

Результат: единый тестируемый доменный слой, общий для web и RN.

## Этап 1 — Порт инфраструктуры

- Реализация `StoragePort`:
  - быстрый старт: `AsyncStorage`;
  - при росте объёма/частоты операций: `MMKV` или SQLite.
- Реализация `TtsPort` на `react-native-tts`.
- Реализация `WakeLockPort` через `expo-keep-awake` (или native Android).

Результат: замещены web API, сохранено поведение домена.

## Этап 2 — Порт UI и навигации

- Экраны переписать на RN по приоритету:
  1. `ScreenSetup`
  2. `ScreenWorkout` (самый критичный и сложный)
  3. `ScreenSettings`
  4. `ScreenDatabase` + detail/about
- Навигацию реализовать через React Navigation:
  - stack для экранов;
  - modal/sheet для деталей упражнения;
  - обработка hardware back с подтверждением отмены тренировки.

Результат: функциональный Android MVP нативного качества.

## Этап 3 — Миграция данных и совместимость

- Продумать импорт данных из веб-версии:
  - JSON import/export (manual migration);
  - или cloud sync в следующей фазе.
- Синхронизировать ключи и схемы данных между web и mobile.

Результат: пользователь не теряет историю и свои упражнения.

## Этап 4 — Качество и релиз

- Перенести критичные unit-тесты домена в shared core.
- Добавить e2e smoke для RN (Detox/Appium минимум на ключевой flow).
- Проверить edge-cases:
  - поведение таймера при background/foreground;
  - прерывания TTS/audio;
  - блокировка экрана и восстановление состояния.

Результат: стабильный релиз для Play Store.

## 6) Ключевые риски и как их снизить

- **Таймеры и lifecycle на Android**: в фоне JS-таймеры могут вести себя иначе.  
  Решение: хранить timestamp начала фазы, считать остаток от системного времени, а не только `setInterval`.

- **TTS-поведение отличается от браузера**: разные движки/голоса/задержки.  
  Решение: абстракция TTS-порта + fallback-сценарии без озвучки.

- **Wake lock/энергосбережение**: устройства и прошивки ведут себя по-разному.  
  Решение: тест на реальных девайсах, явные политики foreground/keep-awake.

- **Миграция local данных**: возможна потеря истории при первом входе в нативное приложение.  
  Решение: встроенный import JSON и UX-подсказки при первом запуске.

## 7) Что можно сделать уже сейчас (без старта полного порта)

- Зафиксировать контракты домена (`types`, интерфейсы портов, тесты).
- Убрать остаточные жёсткие привязки к `window/localStorage` из бизнес-логики.
- Добавить экспорт/импорт пользовательских данных в web-версии.
- Сформировать минимальный Android target scope:
  - только тренировка + настройки + база упражнений;
  - без облака и аккаунтов на первой версии.

Это снизит стоимость и риск последующего перехода на Android независимо от выбранного final-стека.
